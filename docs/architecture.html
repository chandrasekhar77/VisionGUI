CIS-Platform — Architecture Documentation

4-Layer Vision Inspection Platform  ·  MFC / C++  ·  Win32

Overview

CIS-Platform is a 4-layer industrial vision inspection platform. Each layer has a single responsibility and communicates with adjacent layers through well-defined interfaces, so any layer can be replaced independently.

Data flows bottom-up: L1 captures raw hardware signals → L2 processes images and runs algorithms → L3 applies business logic and sequencing → L4 presents results and accepts operator input.

L1

Hardware Abstraction Layer

Camera, light controller, IO devices, and the hardware abstraction engine that normalises all physical I/O.

L2

Image Processing & Algorithm Layer

Image processing engine, rule-based algorithms (OpenCV), AI / deep-learning inference, MIL integration.

L3

Application & Business Layer

Sequence & process controller, pass/fail decision logic, MES / DB / factory server integration.

L4

UI / Management Layer

HMI, dashboard, real-time result display, recipe system, statistics and defect log.

4-Layer Architecture

Sl. No Layer Key Components Description
1 L1 — Hardware Abstraction Layer
  • Hardware Abstraction Engine
  • Camera (GigE / USB3 Vision)
  • Light Controller
  • IO Devices
  • Normalises all physical hardware behind a uniform interface
  • Abstracts camera SDK, lighting protocols, and digital I/O
  • Shields upper layers from hardware vendor differences
  • Connects to L2 via Hardware Abstraction Engine → Image Processing Engine
2 L2 — Image Processing & Algorithm Layer
  • Image Processing Engine
  • Rule-Based Algorithms (OpenCV)
  • AI / Deep Learning Inference
  • MIL / OpenCV Integration
  • Receives raw images from L1 and runs inspection algorithms
  • Rule-based path: classical image processing (thresholding, blob, edge)
  • AI path: deep-learning model inference (classification / detection)
  • Exposes results upward to L3 via Image Processing Engine interface
3 L3 — Application & Business Layer
  • Sequence & Process Controller
  • Pass / Fail Logic
  • MES / DB / Factory Server
  • Drives the full inspection cycle (connect → trigger → inspect → record)
  • Applies tolerance rules and emits final pass / fail verdict
  • Pushes results to MES, factory DB, or external reporting systems
  • Connects to L1 via Hardware Abstraction Engine and to L2 via Image Processing Engine
4 L4 — UI / Management Layer
  • HMI (CMainFrame, CTopBar)
  • Dashboard
  • Real-time Result Display
  • Recipe System
  • Statistics & Defect Log
  • Operator-facing HMI — navigation, controls, status
  • Real-time display of inspection results and defect images
  • Recipe management — create, load, edit, duplicate inspection models
  • Statistics, yield charts, and timestamped defect log
  • Connects to L3 Sequence & Process Controller for commands and results

Multi-Project Reuse

The platform is split into a reusable UI container layer and a project-specific module layer. The table below shows which parts transfer across products unchanged.

Sl. NoComponent / LayerReusable?What changes per project
1L1 UI Container (CMainFrame, CTopBar, CDarkPane, …)Yes — fully reusableNothing. No product knowledge in the UI container.
2IVisionModule interfaceYes — core contractNothing. It is the extension point.
3UITheme.hYesColour constants can be adjusted per brand.
4CImagePanel / D3DImageViewerCtrlYesNothing — generic image + ROI viewer.
5HiResImageRenderer (DLL)YesNothing — generic D3D11 image renderer.
6CLoggingPane / ILoggerYesNothing — generic text log viewer.
7CModelManagerPane / IModelLoaderYesOnly the IModelLoader implementation is product-specific.
8CPerfMonitorYesNothing — generic utility.
9CInspectionModuleNo — PCB specificReplaced entirely with a new IVisionModule for the new product.
10CPCBModel / ROINo — PCB specificReplaced with the product's own model data structure.
11CPanelListPane / DefectInfoPartialSchema and columns are PCB-specific; the concept is reusable.
12CStatsPane / CYieldChartPanePartialChart layout reusable; data fields are product-specific.
13Inspection algorithms (L3)NoReplaced entirely per product.
Summary: L1 UI container, all contract interfaces, the image viewer, and the logging system transfer to a new project with zero changes. Implement IVisionModule (and optionally IModelLoader, IDefectListListener) and register it in CMainFrame::RegisterModules().

Module Independence & Replaceability

The UI container and the module are isolated by IVisionModule. The UI container never #includes a concrete module header — it only knows the interface.

How the UI container loads a module

// CMainFrame — no product knowledge here
void CMainFrame::RegisterModules()
{
    m_modules.push_back(std::make_unique<CInspectionModule>());
    // ← Add any other IVisionModule implementation here
}

void CMainFrame::LoadModule(IVisionModule* pModule)
{
    m_wndTopBar.SetModule(pModule);           // nav tab labels come from module
    for (int i = 0; i < pModule->GetNavCount(); i++)
        m_moduleViews.push_back(pModule->CreateView(i, this, id));
}

What each module must provide

Sl. NoMethodWhat it does
1GetName()Display name shown in the title bar
2GetNavCount()Number of nav tab pills to show in CTopBar
3GetNavLabel(i)Label text for each nav tab pill
4CreateView(i, parent, id)Allocate and return the CWnd for tab i; UI container owns the lifetime
5OnConnect() / OnStart() / OnStop()Called when the top-bar action buttons are clicked
6IsConnected() / IsRunning()State queried by CTopBar to colour the action buttons

Replacing a module

To swap CInspectionModule for a new product (e.g. solder paste inspection):

// 1. Write your module
class CSolderPasteModule : public IVisionModule { ... };

// 2. Register it in MainFrm.cpp — one line change
m_modules.push_back(std::make_unique<CSolderPasteModule>());

// 3. Done — the UI container auto-wires nav tabs, view switching,
//    Connect/Start/Stop, pane visibility, and command routing.

How L4 panes decouple from L2

L4 panes never call L2 directly. L2 posts Windows messages (WM_APP+N) to pane HWNDs; panes update their own display. Panes can be added or reordered without modifying L2.

L2 SequenceController
→
PostMessage(WM_DEFECT_ADDED, lParam=DefectInfo*)
→
CDefectListPane
L2 / any layer
→
VisionLog::Info(_T("msg"))
→
LoggerRegistry
→
CLoggingPane
CTopBar
→
PostMessage(WM_NAV_CHANGED, wParam=navIndex)
→
CMainFrame::OnNavChanged
→
ShowView() + UpdatePaneVisibility()

Dockable Pane Design & Configuration

All side and bottom panels are CDockablePane subclasses managed by the MFC extended docking framework (CFrameWndEx + EnableDocking).

Base class hierarchy

CDockablePane  (MFC framework)
  └── CDarkPane           // dark background; base for all content panes
        ├── CLoggingPane
        ├── CDefectListPane
        ├── CModelManagerPane
        ├── CPanelListPane
        ├── CDefectSummaryPane
        ├── CStatsPane
        ├── CYieldChartPane
        └── CInspectionImagePane

CDockablePane  (MFC framework)
  ├── CTopBar             // CanFloat()=FALSE, CanBeResized()=FALSE — locked to top
  └── CDashboardToolbar   // CanFloat()=FALSE — locked to top, Dashboard tab only

How docking is configured in code

// Image pane — docked to left edge, initial size 380 x 600
m_wndImagePane.Create(_T("Inspection Image"), this,
    CRect(0, 0, 380, 600), TRUE, ID_PANE_IMAGE,
    WS_CHILD | WS_VISIBLE | CBRS_LEFT | CBRS_FLOAT_MULTI);

m_wndImagePane.EnableDocking(CBRS_ALIGN_LEFT | CBRS_ALIGN_RIGHT | CBRS_ALIGN_BOTTOM);
DockPane(&m_wndImagePane, AFX_IDW_DOCKBAR_LEFT);

Locking specific panes

The top navigation bar and dashboard toolbar are locked — cannot be floated or resized:

// In TopBar.h
virtual BOOL CanFloat()     const override { return FALSE; }
virtual BOOL CanBeResized() const override { return FALSE; }

Adding a new dockable pane

// Step 1 — Inherit CDarkPane for consistent dark styling
class CMyPane : public CDarkPane { ... };

// Step 2 — Declare a member in MainFrm.h
CMyPane m_wndMyPane;

// Step 3 — Create and dock in CMainFrame::OnCreate()
m_wndMyPane.Create(_T("My Pane"), this,
    CRect(0, 0, 200, 300), TRUE, ID_PANE_MINE,
    WS_CHILD | WS_VISIBLE | CBRS_RIGHT | CBRS_FLOAT_MULTI);
m_wndMyPane.EnableDocking(CBRS_ALIGN_ANY);
DockPane(&m_wndMyPane);

// Step 4 — Optionally add a visibility rule
m_wndMyPane.ShowPane(m_navIndex == Theme::VIEW_RECIPE, FALSE, FALSE);
Pane state persistence: MFC saves dock positions to the registry via SaveState / LoadState. If you rename a pane ID or change docking flags, the saved state may conflict — clear the registry key or call CMFCVisualManager::ResetToDefaults() during development.
Thread safety: All pane UI updates must happen on the UI thread. Panes receiving data from worker threads (CLoggingPane, CDefectListPane, CPanelListPane) use PostMessage internally to marshal calls before touching any UI element.
