VisionGUI — Platform Architecture

4-Layer Vision Inspection Platform · MFC / C++ · Win32

Overview

VisionGUI is a shell-and-module desktop platform for industrial vision inspection. The shell provides the entire UI frame, docking infrastructure, navigation, and logging. Application-specific logic is encapsulated in interchangeable modules that plug into the shell through a clean C++ interface (IVisionModule).

The platform is divided into four vertical layers. Each layer communicates with the one above and below it through well-defined contracts (pure virtual interfaces and POD structs), so any layer can be replaced or extended without touching the others.

Shell (L1)

MFC frame, dark-themed dockable panes, top navigation bar, window chrome. Knows nothing about inspection.

Module / Sequence (L2)

Active application module implementing IVisionModule. Controls Connect / Start / Stop state machine and owns the inspection sequence.

Vision Engine (L3)

Model data (CPCBModel / ROI), inspection algorithms, image acquisition, DL inference. Fully decoupled from UI.

Data Display (L4)

Individual dockable panes (defect list, logging, stats, image viewer) that react to events posted from L2/L3 via Windows messages.

Multi-Project Reuse

The platform is intentionally split into a reusable shell and a project-specific module. The table below shows which parts transfer across projects unchanged.

PartReusable across projects?What changes per project
L1 Shell (CMainFrame, CTopBar, CDarkPane, …)Yes — fully reusableNothing. The shell has no product knowledge.
IVisionModule interfaceYes — core contractNothing. It is the extension point.
UITheme.h (colors, sizes, WM_APP IDs)YesColor constants can be adjusted per brand.
CImagePanel / D3DImageViewerCtrlYesNothing — generic image+ROI viewer.
HiResImageRenderer (DLL)YesNothing — generic D3D11 image renderer.
CLoggingPane / ILoggerYesNothing — generic text log viewer.
CModelManagerPane / IModelLoaderYesThe IModelLoader implementation is product-specific.
CPerfMonitorYesNothing — generic utility.
CInspectionModuleNo — PCB inspection specificReplaced entirely by a new IVisionModule implementation.
CPCBModel / ROINo — PCB model specificReplaced with product's own model data structure.
CPanelListPane / DefectInfoPartialDefectInfo schema and CPanelListPane columns are PCB-specific; concept is reusable.
CStatsPane / CYieldChartPanePartialChart layout reusable; data fields are product-specific.
Inspection algorithms (L3)NoReplaced entirely per product.
Summary: The entire L1 shell, all contracts, the image viewer, and the logging system transfer to a new project with zero changes. To create a new product, you implement IVisionModule (and optionally IModelLoader, IDefectListListener) and register it in RegisterModules().

Module Independence & Replaceability

The shell and the module are isolated by the IVisionModule interface. The shell never #includes a concrete module header — it only knows the interface.

How the shell loads a module

// CMainFrame — no product knowledge here
void CMainFrame::RegisterModules()
{
    m_modules.push_back(std::make_unique<CInspectionModule>());
    // ← Add any other IVisionModule here
}

void CMainFrame::LoadModule(IVisionModule* pModule)
{
    m_wndTopBar.SetModule(pModule);          // nav labels come from module
    for (int i = 0; i < pModule->GetNavCount(); i++)
        m_moduleViews.push_back(pModule->CreateView(i, this, id));
}

What each module must provide

MethodWhat it does
GetName()Display name shown in the title bar
GetNavCount()How many nav tabs to show in CTopBar
GetNavLabel(i)Label text for each nav tab pill
CreateView(i, parent, id)Allocate and return the CWnd for tab i; shell owns lifetime
OnConnect() / OnStart() / OnStop()Respond to top-bar action buttons
IsConnected() / IsRunning()State queried by CTopBar to colour action buttons

Replacing a module

To swap CInspectionModule for a new product (e.g. a solder paste inspection module):

// 1. Write your module
class CSolderPasteModule : public IVisionModule { ... };

// 2. Register it in MainFrm.cpp — one line change
m_modules.push_back(std::make_unique<CSolderPasteModule>());

// 3. Done — the shell auto-wires nav tabs, view switching,
//    Connect/Start/Stop, pane visibility, and command routing.

How L4 panes decouple from L2

L4 panes never call L2 directly. L2 posts Windows messages to specific pane HWNDs (using the WM_APP+N slots defined in UITheme.h), and panes respond by updating their own display. This means you can add, remove, or reorder panes without modifying L2 at all.

Dockable Pane Design & Configuration

All side and bottom panels are CDockablePane subclasses, managed by the MFC extended docking framework (CFrameWndEx + EnableDocking).

Base class hierarchy

CDockablePane  (MFC)
  └── CDarkPane           // dark background, base for all panes
        ├── CLoggingPane
        ├── CDefectListPane
        ├── CModelManagerPane
        ├── CPanelListPane
        ├── CDefectSummaryPane
        ├── CStatsPane
        ├── CYieldChartPane
        └── CInspectionImagePane

CDockablePane  (MFC)
  ├── CTopBar             // CanFloat()=FALSE, CanBeResized()=FALSE
  └── CDashboardToolbar   // CanFloat()=FALSE

How docking is configured in code

// Example — Image pane docked to left edge, 380px wide
m_wndImagePane.Create(_T("Inspection Image"), this,
    CRect(0, 0, 380, 600), TRUE, ID_PANE_IMAGE,
    WS_CHILD | WS_VISIBLE | CBRS_LEFT | CBRS_FLOAT_MULTI);

m_wndImagePane.EnableDocking(
    CBRS_ALIGN_LEFT | CBRS_ALIGN_RIGHT | CBRS_ALIGN_BOTTOM);

DockPane(&m_wndImagePane, AFX_IDW_DOCKBAR_LEFT);

What the user can do at runtime

  • Drag a pane by its caption to any edge (left, right, bottom) or float it as a window
  • Tab panes together by dropping one pane onto another — they become a tabbed group
  • Resize any pane by dragging the CDarkPaneDivider splitter between panes
  • Minimise a pane to a thin caption strip along the edge to save screen space
  • Close a pane (the × on its caption); it can be restored via ShowPane(TRUE) from code
  • Save and restore layout — MFC serialises dock state automatically via the profile/registry

Locking specific panes

The top navigation bar and dashboard toolbar are locked — they cannot be floated or resized:

// In TopBar.h
virtual BOOL CanFloat()     const override { return FALSE; }
virtual BOOL CanBeResized() const override { return FALSE; }

Adding a new dockable pane

// 1. Create your pane class
class CMyPane : public CDarkPane { ... };

// 2. Declare member in MainFrm.h
CMyPane m_wndMyPane;

// 3. Create + dock in OnCreate()
m_wndMyPane.Create(_T("My Pane"), this,
    CRect(0, 0, 200, 300), TRUE, ID_PANE_MINE,
    WS_CHILD | WS_VISIBLE | CBRS_RIGHT | CBRS_FLOAT_MULTI);
m_wndMyPane.EnableDocking(CBRS_ALIGN_ANY);
DockPane(&m_wndMyPane);

// 4. Optionally add visibility rule in UpdatePaneVisibility()
m_wndMyPane.ShowPane(m_navIndex == Theme::VIEW_RECIPE, FALSE, FALSE);
Pane state persistence: MFC saves dock positions to the Windows registry automatically via SaveState/LoadState. If you rename a pane's ID or change the docking flags, the saved state may conflict — call CMFCVisualManager::ResetToDefaults() or clear the registry key during development.
Thread safety: All pane updates must happen on the UI thread. Panes that receive data from worker threads (CLoggingPane, CDefectListPane, CPanelListPane) use PostMessage internally to marshal from the worker thread to the UI thread before touching any UI element.
